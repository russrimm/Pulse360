Executive Summary

Pulse 360° is a publicly-accessible Next.js 16 application that surfaces Microsoft 365 Message Center data, release plans, security advisories, and product news via Microsoft Graph and assorted RSS feeds. This review examined the source on the russ branch (matches the live mspulse360.app deployment) and produced 20 findings across application code, dependencies, deployment configuration, and operational hygiene.

Single most urgent finding: The Azure AD client secret, tenant ID, and client ID are all stored as NEXT_PUBLIC_* environment variables. Next.js bundles every NEXT_PUBLIC_* value into the client-side JavaScript that ships to the browser. Anyone who visits mspulse360.app can extract these credentials from view-source and use them to authenticate to Microsoft Graph as the application service principal indefinitely, until the secret is rotated. This is the #1 priority remediation item and must be fixed before any other work.

Top 3 things to fix this week

  1. Move all secrets off the client. Rename NEXT_PUBLIC_API_KEY, NEXT_PUBLIC_TENANT_ID, NEXT_PUBLIC_CLIENT_ID to non-public names. Rotate the secret in Azure immediately because it is currently shipping to every visitor.
  2. Lock down or remove /api/proxy-rss. It is a textbook open SSRF — any user can have the server fetch any URL including cloud metadata endpoints.
  3. Apply security headers and run npm audit fix. 4 critical and 19 high CVEs are sitting in node_modules, including Next.js cache poisoning and XSS bypasses.

Detailed Findings

Critical #1 — Azure AD client secret exposed to browser via NEXT_PUBLIC_ prefix

Filessrc/lib/api.ts · .env.local · ..env CWECWE-798 (Hard-coded Credentials), CWE-200 (Information Exposure) Effort~30 minutes + secret rotation

Issue

In Next.js, any environment variable prefixed with NEXT_PUBLIC_ is statically inlined into the client-side JavaScript bundle. The current .env.local stores the Azure AD client secret, tenant ID, and client ID under NEXT_PUBLIC_API_KEY, NEXT_PUBLIC_TENANT_ID, and NEXT_PUBLIC_CLIENT_ID. src/lib/api.ts reads these values, so they end up in any bundle that imports it.

Anyone who visits the deployed site can open browser DevTools → Sources, search for the client ID, and read the secret in plain text. With those three values they can independently mint Graph API tokens as the app's service principal.

Fix

# .env.local (rename ALL three; remove NEXT_PUBLIC_ prefix)
AZURE_CLIENT_ID=32c72b33-...
AZURE_TENANT_ID=f33d7d7f-...
AZURE_CLIENT_SECRET=<new rotated value>

Then in src/lib/api.ts:

const CLIENT_ID = process.env.AZURE_CLIENT_ID;
const TENANT_ID = process.env.AZURE_TENANT_ID;
const API_KEY   = process.env.AZURE_CLIENT_SECRET;

This file is only imported by server-side route handlers (app/api/*/route.ts), so server-only env vars work fine. Verify after the change by running npm run build then grep the .next/static output — the secret must not appear.

Critically: rotate the existing secret in Azure Portal NOW, regardless of whether you ship the rename today. The current secret is in production bundles.

Critical #2 — Open SSRF in /api/proxy-rss

Filesrc/app/api/proxy-rss/route.ts CWECWE-918 (Server-Side Request Forgery) Effort~1 hour

Issue

The route accepts a url query parameter and fetches it server-side with no validation:

const url = searchParams.get('url');
const res = await fetch(url);

This lets any unauthenticated visitor:

  • Probe internal network ranges (10.x, 192.168.x, 172.16-31.x)
  • Hit cloud metadata endpoints (Azure IMDS at http://169.254.169.254/metadata/instance)
  • Read local files via file:// (depending on fetch runtime)
  • Use the server as a relay to mask the attacker's IP when attacking third parties
  • Cause egress traffic that pivots into Vercel/Microsoft infrastructure

Fix

const ALLOWED_HOSTS = new Set([
  'devblogs.microsoft.com',
  'blogs.windows.com',
  'blogs.microsoft.com',
  // ... explicit allowlist
]);

const url = searchParams.get('url');
if (!url) return new NextResponse('Missing url', { status: 400 });

let parsed: URL;
try { parsed = new URL(url); } catch { return new NextResponse('Invalid url', { status: 400 }); }
if (parsed.protocol !== 'https:') return new NextResponse('Only https allowed', { status: 400 });
if (!ALLOWED_HOSTS.has(parsed.hostname)) return new NextResponse('Host not allowed', { status: 403 });

const res = await fetch(parsed.toString(), { redirect: 'manual' });

Even better: replace this route with per-feed routes (you already have most of them). The proxy route appears to exist for arbitrary client-supplied feeds — if no current page actually needs that, delete the route entirely.

Critical #3 — OAuth client-credentials logic in module shared between server and client

Filesrc/lib/api.ts CWECWE-522 (Insufficiently Protected Credentials) Effort~2 hours

Issue

src/lib/api.ts contains the getToken() function that POSTs the client secret to Azure AD. The file is not marked 'use server' and is imported from places that could theoretically be reached client-side. Combined with the NEXT_PUBLIC_* exposure (Finding #1), the entire OAuth client-credentials flow could execute in a browser, leaking the access token to the user.

Fix

  • Add the top-level directive import 'server-only'; at the top of src/lib/api.ts. This causes Next.js to throw a build-time error if the module is ever imported from a Client Component.
  • Once Finding #1 is fixed (env vars no longer NEXT_PUBLIC_*), the runtime cost of accidental client import is also mitigated — the secret won't be available.
  • Move the Graph fetch into a single server-side API route and let client components only call /api/* endpoints, never src/lib/api.ts directly.

Critical #4 — 42 npm vulnerabilities (4 critical, 19 high)

Sourcenpm audit (June 2026) NotableNext.js, ws, lodash, minimatch, xlsx, undici, hono, next-auth Effort~3 hours including testing

Issue

Headline CVEs include:

  • Next.js — cache poisoning via Middleware/Proxy redirects, XSS in App Router with CSP nonces, DoS via Image Optimizer, CSRF bypass via null origin, HTTP request smuggling in rewrites. Multiple CVEs at the installed version.
  • ws (high) — Uninitialized memory disclosure (GHSA-58qx-3vcg-4xpx), DoS via tiny fragments.
  • lodash (high) — Prototype pollution in _.unset/_.omit, code injection via _.template.
  • minimatch (high) — Multiple ReDoS variants via combinatorial backtracking.
  • xlsx (high) — Prototype pollution & ReDoS, no upstream fix available.
  • next-auth (depends on vulnerable uuid) — Buffer bounds check missing.

Fix

  1. Run npm audit fix first — picks up safe non-breaking updates.
  2. Run npm outdated and update Next.js to the latest patch within the same major (currently 16.0.10 → newer 16.x patches).
  3. For xlsx with no fix: either remove the dependency (search for actual usage; it may be unused) or replace with the actively maintained fork @e965/xlsx or exceljs.
  4. For npm audit fix --force changes (will bump next-auth to v3) — review the changelog before accepting; v3 is older not newer. Better to upgrade to next-auth@5 intentionally.
  5. Add npm audit --omit=dev --audit-level=high to CI to block future high/critical CVEs from merging.

High #5 — No global security headers

Filenext.config.js CWECWE-693 (Protection Mechanism Failure) Effort~30 minutes

Issue

next.config.js only configures images.contentSecurityPolicy (which applies only to next/image-served SVGs). The actual application HTML ships with no HSTS, no CSP, no X-Frame-Options, no Referrer-Policy, and no Permissions-Policy. This leaves Pulse 360° open to clickjacking, MIME-sniffing, mixed-content downgrade, and excessive referrer leakage.

Fix — add to next.config.js

async headers() {
  return [{
    source: '/(.*)',
    headers: [
      { key: 'Strict-Transport-Security', value: 'max-age=63072000; includeSubDomains; preload' },
      { key: 'X-Content-Type-Options', value: 'nosniff' },
      { key: 'X-Frame-Options', value: 'DENY' },
      { key: 'Referrer-Policy', value: 'strict-origin-when-cross-origin' },
      { key: 'Permissions-Policy', value: 'camera=(), microphone=(), geolocation=()' },
      { key: 'Content-Security-Policy', value:
        "default-src 'self'; " +
        "script-src 'self' 'unsafe-inline' va.vercel-scripts.com vitals.vercel-insights.com; " +
        "style-src 'self' 'unsafe-inline'; " +
        "img-src 'self' data: https:; " +
        "font-src 'self' data:; " +
        "connect-src 'self' https://graph.microsoft.com https://login.microsoftonline.com va.vercel-scripts.com vitals.vercel-insights.com; " +
        "frame-ancestors 'none'; " +
        "base-uri 'self'; " +
        "form-action 'self';" },
    ],
  }];
},

Tune connect-src to match all upstream RSS hosts the client side actually hits. Test on a deployment preview before merging.

High #6 — dangerouslyAllowSVG: true

Filenext.config.js (line 6) CWECWE-79 (XSS), CWE-434 (Unrestricted Upload) Effort~1 hour (audit current SVG usage)

Issue

SVG files can contain <script> tags and event handlers. dangerouslyAllowSVG opens the door even with the per-image CSP. The repo bundles ~500 Microsoft Azure icon SVGs locally and references remote SVGs (e.g. AI Foundry favicon from devblogs.microsoft.com) — if any remote SVG is ever attacker-controlled, this becomes XSS.

Fix

Either (a) remove dangerouslyAllowSVG: true and switch to inline SVG components (the repo already vendors them locally so this is feasible), or (b) keep it but tighten the per-image CSP and document an SVG-vetting policy. Option (a) is recommended for new images.

High #7 — Unsanitized HTML rendering in 8 components

FilesMessageDetail.tsx, ReleasePlanDetail.tsx, AzureUpdateCard.tsx, M365UpdateCard.tsx, ReleasePlanCard.tsx, plus 3 page.tsx files CWECWE-79 (XSS) Effort~2 hours

Issue

9 occurrences of dangerouslySetInnerHTML across 8 files render HTML returned from Graph API, RSS feeds, and external blog content with no sanitization. isomorphic-dompurify is already a project dependency — but is never imported anywhere in src/.

Graph API content is high-trust today, but RSS feeds are not — anything devblogs.microsoft.com/foundry publishes ends up here unfiltered. A compromised author or a publishing bug upstream becomes site-wide XSS.

Fix

Create src/lib/sanitize.ts:

import DOMPurify from 'isomorphic-dompurify';
export const cleanHtml = (html: string) =>
  DOMPurify.sanitize(html, {
    ALLOWED_TAGS: ['p','b','i','em','strong','a','ul','ol','li','br','h1','h2','h3','h4','code','pre','blockquote','img','table','thead','tbody','tr','th','td'],
    ALLOWED_ATTR: ['href','target','rel','src','alt','title','class'],
    ALLOW_DATA_ATTR: false,
  });

Then replace every dangerouslySetInnerHTML={{ __html: x }} with dangerouslySetInnerHTML={{ __html: cleanHtml(x) }}.

High #8 — No rate limiting on any API route

Files18 routes under src/app/api/ CWECWE-770 (Allocation of Resources Without Limits) Effort~2 hours

Issue

Anyone can hammer /api/messages to exhaust the Graph API quota (incurring real Azure cost / tenant-wide throttle) or /api/proxy-rss to relay traffic. There is no per-IP or per-route rate limit, and no middleware.

Fix

Add Vercel's @upstash/ratelimit + Upstash Redis (free tier) and create middleware.ts at the project root applying limits to /api/*:

// middleware.ts
import { Ratelimit } from '@upstash/ratelimit';
import { Redis } from '@upstash/redis';
import { NextRequest, NextResponse } from 'next/server';

const limiter = new Ratelimit({
  redis: Redis.fromEnv(),
  limiter: Ratelimit.slidingWindow(30, '1 m'),
});

export async function middleware(req: NextRequest) {
  const ip = req.headers.get('x-forwarded-for') ?? 'anon';
  const { success } = await limiter.limit(ip);
  if (!success) return new NextResponse('Too Many Requests', { status: 429 });
  return NextResponse.next();
}
export const config = { matcher: '/api/:path*' };

High #9 — All 18 API routes are unauthenticated

FilesEvery route.ts CWECWE-862 (Missing Authorization) Effort~4 hours (depends on auth model decision)

Issue

None of the API routes check who the caller is. The data is mostly publicly-published Microsoft content, so this is acceptable for read-only feeds, but it also means the routes are CDN-bypass attack vectors and there's no future-proofing for any user-specific feature.

Fix

  • For routes that are intentionally public (RSS proxies of public feeds): rate-limit aggressively (Finding #8) and add caching headers.
  • For /api/messages (real Graph API call costing tenant quota): require a session cookie or a referer/origin check at minimum. Long term, add Entra ID sign-in via NextAuth and gate the route to authenticated users.

Medium #10 — Misnamed ..env file with real secret not gitignored

File..env (note: leading double dot) Effort~5 minutes

Issue

A file literally named ..env (double dot) sits in the repo root containing the old client secret. .gitignore covers only .env and .env*.local, so ..env would be tracked if anyone ever runs git add ..

Fix

  1. Delete ..env (the secret in it is rotated anyway after Finding #1).
  2. Tighten .gitignore to *.env* (catches all variants) and explicitly allow .env.example.
  3. Add a pre-commit hook with git-secrets or gitleaks to block credential commits.

Medium #11 — Error responses disclose env var names

Filesrc/app/api/messages/route.ts CWECWE-209 (Information Exposure Through Error Message)

Issue

On 500, the route returns a JSON payload including a hint that names NEXT_PUBLIC_TENANT_ID, NEXT_PUBLIC_CLIENT_ID, and NEXT_PUBLIC_API_KEY. The actual upstream error text is also forwarded verbatim (currently leaking Azure AAD trace IDs, correlation IDs, and AADSTS error codes). This helps attackers reconnaissance the auth setup.

Fix

Return a generic { error: 'Upstream service unavailable' } to the client. Log the detailed error server-side only (via console.error or your observability tool). Keep the rich diagnostics out of the HTTP body.

Medium #12 — No CSRF protection (footgun for future state-changing endpoints)

Issue

All current API routes are GETs returning data, so CSRF is not exploitable today. The risk is forward-looking: the first POST/DELETE route added (e.g., subscription endpoint, comment, bookmark) will be CSRF-vulnerable by default in Next.js App Router.

Fix

Adopt a CSRF strategy now: either use NextAuth's built-in CSRF token, or add a SameSite=Strict session cookie + Origin header check in middleware.ts.

Medium #13 — Path-injection in /api/msrc?monthId=

Filesrc/app/api/msrc/route.ts

Issue

const res = await fetch(CVRF_URL + monthId) directly concatenates user input. monthId=../../v1.0/foo rewrites the path. The blast radius is limited to api.msrc.microsoft.com, but still allows hitting non-CVRF endpoints on that host.

Fix

Validate against a strict regex: if (!/^\d{4}-[A-Za-z]{3}$/.test(monthId)) return 400; (MSRC month IDs are 2026-Jun shape). Use encodeURIComponent(monthId) when building the URL.

Medium #14 — No CORS policy declared

Issue

API routes return no Access-Control-Allow-Origin header. Browsers fall back to same-origin, so it appears safe — but if anyone later adds Access-Control-Allow-Origin: * reflexively, the now-public routes (Finding #9) become available to any third-party site, including expensive Graph-backed ones.

Fix

Decide explicitly: either reject cross-origin (default; document it) or allowlist specific origins. Add a unit-test that fails if Access-Control-Allow-Origin: * appears on any /api/* response.

Medium #15 — No documented secret rotation policy

Issue

The Azure AD client secret is created with multi-year expiry in Azure and stored statically. There is no calendar reminder, no rotation runbook, and no detection if it leaks. The secret rotated during this review is good for ~2 years and then will silently expire.

Fix

  • Store secrets in Vercel project env settings (or Azure Key Vault for non-Vercel environments) with notes on expiry.
  • Set a calendar reminder 30 days before expiry.
  • Better: migrate from client-secret to Federated Identity Credentials (Workload Identity Federation) — no secret to rotate. Azure supports FIC for GitHub Actions and Vercel today.

Medium #16 — Production silences all console.* methods

Filesrc/lib/api.ts lines 4–10

Issue

if (process.env.NODE_ENV === 'production') {
  console.log = function() {};
  console.warn = function() {};
  console.error = function() {};
  ...
}

This pattern blanket-mutes every log on the production server, including security-relevant errors. When something goes wrong in production you have no breadcrumbs.

Fix

Remove the override entirely. Use Vercel's built-in log aggregation or wire up Sentry / Application Insights. Keep console.error active at minimum.

Low #17 — xlsx package with no upstream fix

The xlsx dependency has known prototype pollution and ReDoS with no patch available. If it's actually used (search the codebase — appears to be unused at first glance), replace with exceljs or @e965/xlsx (the maintained fork). If unused, remove from package.json.

Low #18 — No Subresource Integrity on remote assets

External images loaded via next/image from devblogs.microsoft.com have no SRI. If Microsoft's CDN is ever compromised, you serve compromised content. Not exploitable today; document as accepted risk or vendor the few external images locally.

Low #19 — Image remotePatterns uses pathname: '**'

Allowing ** on devblogs.microsoft.com means any path on that host can be requested through /_next/image. Combined with dangerouslyAllowSVG this widens the attack surface. Narrow to specific path prefixes you actually use.

Low #20 — Generic try/catch swallows root cause

Several API routes catch any error and return 'Error fetching feed' or similar. The actual error is lost. Replace with console.error('proxy-rss failed', { url, err }) at minimum, or wire to Sentry.
